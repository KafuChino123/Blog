#### five. Obj-C typedef & Swift typealias

In Objective-C and Swift, the core purpose of `typedef` and `typealias` is similar, both of which are defining aliases for existing types to improve code readability and simplify the declaration of complex types, but they have some differences in function and flexibility. The following is a detailed comparison:

##### **Common points**

1. **Basic alias function**

- `typedef` of **Objective-C** and `typealias` of **Swift** can define aliases for existing types (basic types, structures, classes, etc.).

- For example:

```Objective-C

// Objective-C

Typedef int MyInt;

Typedef void (^CompletionBlock)(void);

```

````swift

// Swift

Typealias MyInt = Int

Typealias CompletionBlock = () -> Void

```

Note: `^` means that this is a **[[##^439e1a|Block type]]**, similar to a function pointer, but more powerful (can capture context variables).

##### **Difference**

1. **General support**

- **Swift**'s `typealias` can directly support generic aliases, and even ** partially specify generic parameters**:

````swift

Typealias StringDictionary<T> = Dictionary<String, T>

Typealias IntArray = Array<Int>

```

- **Objective-C**'s `typedef` does not support generic parameterization. Although Objective-C has lightweight generics (such as `NSArray<NSString *>`), generic alias cannot be created through `typedef`.

2. **Agreement combination**

- **Swift**'s `typealias` can be used to combine multiple protocols or add protocol constraints to the type:

````swift

Protocol A {}

Protocol B {}

Typealias ABProtocol = A & B // Follow the protocol combination of A and B at the same time

Typealias ViewWithAB = UIView & A & B // Combination of types and protocols

```

- The `typedef` of **Objective-C** cannot realize the protocol combination (the protocol of Objective-C itself supports the combination syntax `id<ProtocolA, ProtocolB>`, but cannot be simplified by the alias `typedef`).

3. **Associated Type**

- In the **Swift** protocol, `typealias` can be used as an alternative to association types (such as simplifying the implementation of generic protocols):

````swift

Protocol container{

Associated type item

Func add(_ item: Item)

}

Class IntContainer: Container {

Typealias Item = Int // Implement the associated type

Func add(_ item: Int) { ... }

}

```

- **Objective-C** has no similar functions, because its generic and protocol system are relatively simple.

4. **Scape and expandability**

- **Swift**'s `typealias` can be defined in global, class, structure, enumeration and even within the protocol, which is more flexible.

- The `typedef` of **Objective-C** is usually global or defined in the header file, and the scope control is weak.

##### **Summary**

| Characteristics | Objective-C `typedef` | Swift `typealias` |

|--------------------|-----------------------------|--------------------------------|

| **Basic type alias** | ✅ | ✅ |

| **CLOSER/FUNCTUAL POINTER ALIAS** | ✅ | ✅ |

| ** generic alias** | ❌ | ✅ (support parameterization) |

| **Protocol combination** | ❌ | ✅ |

| **Association type** | ❌ | ✅(in the agreement) |

| **Scope flexibility** | Weak | Strong (groball, internal type, extension, etc.) |

##### **Applicable scenario**

- **Objective-C `typedef`**: The name suitable for simplifying the base type, structure, enumeration or closure type.

- **Swift `typealias`**: In addition to the basic alias, it can also be used for protocol combination, generic optimization and protocol design, which is a more modern and feature-rich tool.

If you need to migrate Objective-C's `typedef` in Swift, you can directly replace it with `typealias` in most cases, but Swift's aliases are more flexible and powerful.

##### 6.6 Informal Agreement

- Informal agreement: essentially a classification, listing a set of methods, but **not implementing these methods**. Everyone (or almost everyone) inherits the same root object. Therefore, == informal classification is usually defined for the root class ==. Sometimes, informal agreements are also called abstract agreements.

- The class that declares the informal protocol does not implement these methods itself, and the subclass that chooses to implement these methods needs to re-declare these methods in its interface part, and also implement one or more of these methods. Unlike formal protocols, compilers do not provide help with informal protocols. - There is no concept of compliance with protocols or testing by compilers.

- If an object adopts a formal agreement, it must comply with all the information in the agreement. This can be enforced during operation and compilation. If an object adopts an informal agreement, it may not need to adopt all the methods of this agreement (depending on the agreement). You can enforce compliance with an informal protocol at runtime (with `respondsToSelector:`), but not at compile time.

- Note: The `@optional` instruction has been added to the Objective-C 2.0 language to replace the use of informal protocols. We can see that several UIKit classes are used in this way.

##### 6.7 Synthetic Objects (Combination vs Inheritance)

- There is such a class that contains one or more objects of other classes. The object of this new class is == composite ==, because it is composed of other objects.

For example, for the Square class, it is natural to define this class as a subclass of Rectangle, because a square is an equilateral rectangle. When defining a subclass, it inherits all the instance variables and methods of the parent class. But in some cases, this practice is not good - because ** some methods defined in the parent class may not be suitable for subclass use **. The Square class inherits the `setWidth:andHeight:` method of Rectangle to set the width and height, but it is not applicable to Square (although it can work normally). In addition, when creating subclasses, it is necessary to ensure that all inherited methods can work properly, because users of this class may access them. In the end, the subclass depends on the parent class, and changing the parent class may prevent the methods in the subclass from working properly.

As an alternative to creating subclasses, a new class can be defined, which contains the instance variables to be extended to the class. Then, just define the method suitable for the class in the new class. Return the example of Square. The following is another definition of the Square class:

```Objective-C

@Interface Square: NSObject {

Rectangle *rect;

}

-(Int) setSide: (int)s;

-(Int) side;

-(Int) area;

-(Int) perimeter;

@End

@Interface Rectangle: NSObject

@Property int width, height;

- (void) setWidth: (int) w andHeight: (int) h;

- (int) area;

- (int) perimeter;

@End

```

The Square class defined here has 4 methods. Unlike the subclass version, the subclass version allows you to directly access the methods of Rectangle (`setWidth:andHeight:`, etc.). At the same time, the definition of Square does not include these, and it is not suitable for methods directly applied to Square.

If Square is defined in this way, you need to allocate storage space for the rectangle it contains. If the method is not rewritten, the following statement

```Objective-C

Square *mySquare = [[Square alloc] init];

```

A new Square object will be assigned, but no storage space will be allocated for the Rectangle object rect stored in the instance variable (here means that the ==`rect` pointer does not point to any valid `Rectangle` instance (not the pointer itself is not allocated) ==). Therefore, it may be necessary to rewrite `init` or add new methods such as `initWithSide:` to allocate space and set its edges accordingly.

When defining methods in the Square class, you can also use the method of Rectangle:

```Objective-C

//Square.m

@Implementation Square

- (instancetype)init {

Self = [super init];

If (self) {

Rect = [[Rectangle alloc] init]; // explicitly allocate memory

}

Return self;

}

-(Int) area {

Return [rect area];

}

@End

```

- **Inheritance and combination, the difference in memory allocation:**

- **Inheritance**:

As a subclass of `Rectangle`, `Square` inherits its `width` and `height` variables. When creating a `Square` instance, the storage space of the parent class `Rectangle` ** is automatically included in the subclass ** without additional allocation.

- **Combination**:

`Square`== only holds the pointer `rect`== of `Rectangle`, and must == manually manage the life cycle of `rect`==. If memory is not allocated at the time of initialization, `rect` will not be available.

- ==**The combination mode requires explicit management of the memory of member objects**==, which is a core difference compared with inheritance.

#### 8. Blocks (Blocks)

^439e1a

Block is an extension of C language. It is not defined as a part of the standard ANSI C, but is added to the language by Apple. The block looks more like a function. Parameters can be passed to the block, just like to the function; and the block also has a return value. Unlike functions, == blocks are defined inside the function or method, and can access any variable within the scope of the function or method and outside the block ==. Generally speaking, == these variables can be accessed but cannot change the value of these variables ==.

There is a special modifier (composed of characters with two underscores in front of the block) that can modify the values of variables in the block.

Blocks can == pass to functions or methods as parameters ==, and one of the advantages of blocks is that == can allow the system to execute other threads assigned to other processors or applications ==.

For example, about this `printMessage:` function:

```Objective-C

Void printMessage (void){

NSLog(@"Programming is fun.");

}

```

Such a Block can also complete the same task:

```Objective-C

^(void){

NSLog(@"Programming is fun.");

}

```

The block is marked with the beginning of the inserted character `^`. The following parentheses indicate the list of parameters required by the block. In this example, the block has no parameters, so you only need to fill in void in the function definition.

Similarly, this block can also be assigned to a variable called `printMessage`, as long as the variable is declared correctly:

```Objective-C

Void (^printMessage) (void) =

^(void){

NSLog(@"Programming is fun.");

}

```

The left side of the equal sign indicates that `printMessage` points to a block pointer without parameters and return values. ** It should be noted that the assignment statement is terminated by a semicolon. **

Execute a block referenced by a variable, which is consistent with the call method of the function:

```Objective-C

Print message();

```

Blocks can access variables defined within its scope. The value of the variable is also the value defined in the block, as follows:

```Objective-C

#Import <Foundation/Foundation.h>

Int main (int argc, char *argv[]){

@Autoreleasepool{

Int foo = 10;

Void (^printFoo) (void) =

^(void){

NSLog(@"foo = %i", foo);

};

Foo=15;

Printfoo();

}

Return zero;

}

```

Output:

```

Foo=10

```

Block printFoo can access the value of the local variable foo. == Note that the value of foo shows 10, not 15. This is because the variable foo already has a value at the same time as **defining the block**, not when the block is executed. ==

==The value of the defined variable cannot be modified inside the block==. Therefore, if you try to change the value of foo inside the block, you will get an error message prompted by the compiler: assign a value to the read-only variable "foo".

#Import <Foundation/Foundation.h>

If you insert the `__block` modifier before defining the local variable foo:

```Objective-C

__Block int foo = 10;

```

Run the program, and you will get two output statements:

```

In block: foo = 15

Out of block: foo = 20

```

The first line shows the value of foo when calling the block, and the second line verifies whether the foo value can be changed to 20 in the block.

##### 8.1 About `__block` modifiers

In Objective-C, using the `__block` modifier will change the way the block captures external variables, so that the modification of variables inside and outside the block can be synchronized. Specifically:

1. **Default behavior (no `__block` modifier)**:

- The block will capture the **instantaneous value** (value copy) of the external variable, and the subsequent modification of the external variable will not affect the value inside the block.

- As in the example, `foo` is changed to 15 after the block is defined, but the block still outputs the value of 10 at the time of definition.

2. **Use `__block` modifier**:

- Variables will be captured by **reference** instead, and the same storage location will be shared inside and outside the block.

- What is accessed in the block is the **real-time value** of the variable, not the snapshot at the time of definition.

- At this time, if the block is called after externally modifying the variable, the block will use the latest value.
