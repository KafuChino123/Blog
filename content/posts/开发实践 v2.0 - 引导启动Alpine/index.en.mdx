# Development Practice v2.0 - Boot Alpine

> This document summarizes the differences, problems and solutions encountered when running Ubuntu and Alpine Linux using Direct Kernel Boot in Apple Virtualization Framework.

##1. Brief account

### 1.1 What is direct kernel booting?

**Direct Kernel Boot** is a way to start a virtual machine that bypasses traditional bootloaders (such as GRUB, systemd-boot). Hypervisor directly loads the Linux kernel into memory and executes it, instead of starting a bootloader first and then loading the kernel.

```

┌──────────────────────────────────────────────┐

│ Traditional guidance process │

├──────────────────────────────────────────────────┤

│ UEFI/BIOS → Bootloader (GRUB) → kernel → initramfs → init │

└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐

│ Direct kernel boot process │

├──────────────────────────────────────────────────┤

│ Hypervisor → Kernel → initramfs → init │

└──────────────────────────────────────────────┘

```

**Advantages**:

- ⚡ Faster startup (no bootloader stage)

- 🔧 More flexible configuration (kernel parameters can be dynamically modified on the host machine)

- 🐛 Debugging is more convenient (can easily replace the kernel for testing)

**Disadvantage**:

- Need to manually manage kernel and initramfs files

- After the client kernel is updated, the boot configuration of the host needs to be updated synchronously.

### 1.2 Necessary components

Apple Virtualization Framework's `VZLinuxBootLoader` requires the following components to realize direct kernel booting:

| Components | Description | Is it necessary |

|-------|-------|:--------:|

| **vmlinux / Image** | Uncompressed Linux kernel binary file | ✅ |

| **initrd / initramfs** | Initial memory file system, including drivers and tools required for early startup | ⚠️ Usually required |

| **root filesystem** | Root file system disk image (raw or other formats) | ✅ |

| **kernel command line** | Kernel startup parameters (such as `root=`, `console=`, etc.) | ✅ |

> **💡 Note**: `VZLinuxBootLoader` is a Swift/Objective-C API provided by Apple Virtualization Framework, which is used to configure the direct kernel boot of the Linux virtual machine. It accepts kernel file path, initrd path and command line parameters as input.

## three. Problems and solutions encountered

### 3.1 Kernel format is incompatible

#### phenomenon

```

Internal Virtualization error. The virtual machine failed to start.

```

#### Cause Analysis

The `vmlinuz-virt` of Alpine ARM64 is in EFI stub format:

"Bash"

$ file vmlinuz-virt

Vmlinuz-virt: PE32+ executable (EFI application) Aarch64

```

Initially, it was mistaken that the EFI stub format could not be used for direct kernel booting, but in fact **Ubuntu's kernel is also EFI stub format**:

"Bash"

$ file ~/LinuxVM/vmlinux

Vmlinux: Linux kernel ARM64 boot executable Image, little-endian, 4K pages

```

The hexadecimal head structure of both is the same:

```

The meaning of offset hexadecimal ASCII

────────────────────────────────────────

00000000 4d5a 40fa ... MZ@... DOS/PE header magic

00000038 4152 4d64 ... ARMd ARM64 magic number

00000040 5045 0000 ... PE.. PE header signature

```

**The real problem**: The `vmlinuz-virt` of Alpine netboot embeds the gzip compressed kernel into the EFI stub, which needs to be extracted and decompressed first.

#### Solution

**Step 1**: Locate and extract gzip data from EFI stub

```python

#! /Usr/bin/env python3

"""Extract gzip compressed kernel from EFI stub kernel""""

Import sys

Def extract_kernel(input_file, output_file):

With open(input_file, 'rb') as f:

Data=f.read()

# Find gzip magic number (1f 8b 08)

Gzip_magic=b'\x1f\x8b\x08'

Gzip_offset=data.find(gzip_magic)

If gzip_offset == -1:

print("Error: gzip magic not found")

Sys.exit(1)

Print(f"Found gzip data at offset: {gzip_offset} (0x{gzip_offset:x})")

With open(output_file, 'wb') as f:

F.write(data[gzip_offset:])

Print(f"Extracted to: {output_file}")

If __name__ == "__main__":

Extract_kernel('vmlinuz-virt', 'kernel.gz')

```

**Step 2**: Uncompress the kernel

"Bash"

Gunzip -c kernel.gz > vmlinux

Vmlinux file

# Expected output: Linux kernel ARM64 boot executable Image, little-endian, 4K pages

```

### 3.2 Disk mirror download failed (silent failure)

#### phenomenon

After booting, the kernel reported that the root file system could not be mounted, and the disk image was found to be empty or the size was abnormal.

#### Cause Analysis

The cloud mirror URL in the download script returns **HTTP 404**, but `curl` does not use the `-f` parameter, resulting in the saving of the error page as a "mirror file":

"Bash"

# ❌ Wrong URL (version number does not match, return 404)

CLOUD_IMAGE_URL="https://dl-cdn.alpinelinux.org/.../nocloud_alpine-3.21.2-aarch64.qcow2"

# ✅ Correct URL

CLOUD_IMAGE_URL="https://dl-cdn.alpinelinux.org/.../nocloud_alpine-3.21.0-aarch64-uefi-tiny-r0.qcow2"

```

#### Solution

Use a secure download method and verify the results:

"Bash"

# Use the -f parameter to let curl return the failure status code when HTTP error

Curl -fL -o alpine.qcow2 "$CLOUD_IMAGE_URL"

# Verify the download content

File alpine.qcow2

# Expected output: QEMU QCOW2 Image (v3), 562036736 bytes

# Or check the file size (Alpine cloud mirror is usually > 40MB)

Ls -lh alpine.qcow2

```

> **💡 Best Practices**: Always verify the downloaded mirror image with checksum. Alpine officially provides `.sha256` or `.sha512` verification documents.

### 3.3 GPT backup header location error

#### phenomenon

Output warning when the kernel starts:

```

GPT:Primary header thinks Alt. header is not at the end of the disk.

GPT:335871! = 8388607

GPT:Alternate GPT header not at the end of the disk.

GPT: Use GNU Parted to correct GPT errors.

```

#### Cause Analysis

The GPT (GUID Partition Table) partition table adopts a redundant design to store a partition table at the *head and tail** of the disk:

```

Expansion front disk (164 MB) Expansion back disk (4 GB)

┌────────────────┐┌────────────────┐

│ Primary GPT Header │ │ Primary GPT Header │ ← Point to the position 335871

├──────────────────┤├────────────────┤

││││

│ Partitions │ │ Partitions │

││││

├────────────────┤││

│ Backup GPT Header │← Location 335871 │ │

└──────────────────┘ │ (New Space) │

││

├────────────────┤

│??? (There should be a backup header) │ ← Location 8388607

└──────────────────┘

```

After using `qemu-img resize` to expand the disk, the backup head is still in its original location (335871), not at the end of the new disk (8388607).

#### Solution

Use `sgdisk` to repair GPT metadata:

"Bash"

# Install sgdisk (macOS)

Brew install gptfdisk

# Repair GPT: Move the backup header to the end of the disk

Sgdisk -e alpine.raw

# Verify the repair results

Sgdisk -v alpine.raw

# Expected output: No problems found.

```

> **📚 Extended knowledge**: `-e` of `sgdisk -e` means "move backup to end", which will:

> one. Read the main GPT header

> two. Rewrite the backup GPT header at the end of the new disk

> three. Update the backup head position pointer in the main header

### 3.4 Root partition position error

#### phenomenon

```

VFS: Cannot open root device "/dev/vda1" or unknown-block(254,1): error -6

```

Someone

```

Mount: mounting /dev/vda1 on /sysroot failed: No such file or directory

```

#### Cause Analysis

The cloud mirror image of different distributions adopts different partitioning strategies:

| Distribution | Partition Scheme | Root Partition Equipment |

|--------|---------|------------|

| Ubuntu Cloud Mirror | MBR, Single Partition | `/dev/vda1` |

| Alpine Cloud Mirror (UEFI) | GPT, Dual Partition | `/dev/vda2` |

| Alpine network installation (default) | No partition (whole disk) | `/dev/vda` |

#### Solution

Configure the correct `root=` parameter according to the actual partition situation:

"Bash"

# Check the disk partition (need to mount or use fdisk check first)

Fdisk -l Alpine.raw

#Ubuntu

Root=/dev/vda1

# Alpine Cloud Mirror Image (UEFI)

Root=/dev/vda2

# Alpine network installation to the whole disk (no partition table)

Root=/dev/vda

```

> **💡 Tip**: You can use `PARTUUID` or `UUID` instead of the device name, which is more stable and reliable:

>````

> root=PARTUUID=xxxxxxxxx-xxxx-xxxx-xxxxxxxxxxxx

>````

### 3.5 Cloud mirroring is not compatible with netboot kernel/initramfs

#### phenomenon

Even if the GPT has been repaired and the partition location is correct, the root file system cannot be mounted:

```

Mount: mounting /dev/vda2 on /sysroot failed: Invalid argument

```

#### Cause Analysis

Alpine cloud mirror (`nocloud_alpine-*-uefi-*`) is a complete system designed for **UEFI boot**:

1. **The kernel/initramfs version does not match**: The built-in kernel module version of cloud mirroring may be inconsistent with the netboot kernel version.

2. **initramfs function difference**: netboot's initramfs is designed for network installation and does not contain the logic required to mount the pre-installed system.

3. **Special mounting options**: The root file system of cloud mirroring may use specific mounting options or overlay configuration.

#### Solution

**Recommended plan**: Use the network installation mode to install Alpine on a blank disk

"Bash"

#1. Create a blank disk image

Qemu-img create -f raw alpine-fresh.raw 4G

Number two. Download the netboot file

ALPINE_VERSION="3.21"

BASE_URL="https://dl-cdn.alpinelinux.org/alpine/v${ALPINE_VERSION}/releases/aarch64/netboot"

curl-fLO "${BASE_URL}/vmlinuz-virt"

curl-fLO "${BASE_URL}/initramfs-virt"

#Three. Extract and decompress the kernel (see Section 3.1)

#Four. Use the network installation command line to start

```

**The kernel command line of network installation**:

```

Console=hvc0 ip=dhcp alpine_repo=https://dl-cdn.alpinelinux.org/alpine/v3.21/main modules=loop,squashfs,virtio_blk,virtio_net

```

| Parameters | Description |

|-----|-----|

| `console=hvc0` | Use the virtio console (Apple Virtualization Framework uses this device) |

| `ip=dhcp` | Get network configuration through DHCP |

|`alpine_repo=... ` | Specify the address of the Alpine software package warehouse |

|`modules=... ` | List of preloaded kernel modules |

**After the installation is completed**, modify the command behavior:

```

Console=hvc0 root=/dev/vda rw

```

## five. In-depth technical details

### 5.1 EFI Stub Kernel

The modern Linux ARM64 kernel is compiled by default in **EFI stub** format (enabled by `CONFIG_EFI_STUB=y`), with dual identity:

| Role | Description |

|-----|-----|

| **PE32+ executable file** | Can be directly loaded and executed by UEFI firmware |

| **Traditional Linux kernel** | Can be loaded by bootloader or hypervisor |

**The internal structure of the EFI stub kernel**:

```

┌────────────────────────────────────┐

│DOS MZ Header (64 bytes)│

│-Magic: "MZ" (0x4D5A)│

│-PE header offset pointer│

├──────────────────────────────────────┤

│PE/COFF Header│

│- Signature: "PE\0\0" (0x50450000) │

│- Machine: 0xAA64 (ARM64) │

│ - Characteristics: executable, no relocs │

├──────────────────────────────────────┤

│ ARM64 Linux Header (starting from 0x38 offset) │

│-Magic: "ARMd" (0x644D5241)│

│ - Image size, load offset, flags... │

├──────────────────────────────────────┤

│Compressed Kernel Data (gzip/lz4/etc.) │

│ - Start with 0x1F 0x8B 0x08 (gzip) │

└────────────────────────────────────┘

```

### 5.2 The role of Initramfs

**Initramfs (Initial RAM Filesystem)** is a temporary root file system that provides the necessary environment before the real root file system is available.

#### Why does direct kernel boot need initramfs?

```

┌──────────────────────────────────────────────────────┐

│ Kernel start │

│↓│

│ Virtio_blk driver is required to access virtual disk │

│↓│

│ virtio_blk is compiled into a module (.ko) and stored on the disk /lib/modules/ │

│↓│

│ ❌ Deadlock: You need a driver to read the disk, and you need to read the disk to load the driver │

└────────────────────────────────────────────────────┘

Solution: Initramfs

┌──────────────────────────────────────────────────────┐

│ Bootloader/Hypervisor loads initramfs into memory │

│↓│

│ Kernel mount initramfs as a temporary root file system │

│↓│

│ Load virtio_blk.ko module from initramfs │

│↓│

│ Now you can access the virtual disk │

│↓│

│ Mount the real root file system, pivot_root switch past │

└────────────────────────────────────────────────────┘

```

**Typical error without initramfs**:

```

VFS: Cannot open root device "/dev/vda2" or unknown-block(0,0): error -19

Please append a correct "root=" boot option

Kernel panic - not syncing: VFS: Unable to mount root fs on unknown-block(0,0)

```

> **📝 Description**: `error -19` corresponds to the Linux error code `ENODEV` (No such device), indicating that the kernel cannot recognize the virtio block device.

### 5.3 GPT vs MBR Detailed Comparison

| Features | MBR (Master Boot Record) | GPT (GUID Partition Table) |

|-------|------------------------|----------------------------|

| **Introduction time** | 1983 (IBM PC) | 2000s (UEFI specification) |

| **Maximum disk size** | 2 TiB (2^32 × 512B) | 9.4 ZB (2^64 × 512B) |

| **Maximum number of partitions** | 4 primary partitions | 128 partitions (extensible) |

| **Partition table location** | Only at the beginning of the disk | Beginning + end (redundant backup) |

| **Partition Identification** | 1-byte type code | 128-bit GUID |

| **Data Integrity** | No Verification | CRC32 Verification |

| **UEFI boot** | Requires compatibility mode | Native support |

**GPT partition table structure**:

```

LBA 0: Protective MBR (compatible with old tools)

LBA 1: Main GPT header (including backup header location pointer)

LBA 2-33: Main partition table entries (128 bytes per entry)

LBA 34-...: Partition Data

...

LBA -33 to -2: Backup partition table entries

LBA -1: Backup GPT header

```

### 5.4 VirtIO Introduction

**VirtIO** is a standardized virtual device interface that allows virtual machines to communicate efficiently with the host using semi-virtualized drivers.

| VirtIO Device | Function | Device Node Example |

|------------|------|-------------|

| `virtio_blk` | Block storage device | `/dev/vda`, `/dev/vdb` |

| `virtio_net` | Network interface | `eth0`, `enp0s1` |

| `virtio_console` | console/serial port | `/dev/hvc0` |

| `virtio_rng` | Random number generator | `/dev/hwrng` |

| `virtio_balloon` | Memory balloon (dynamic adjustment) | - |

> **💡 Why is it `/dev/hvc0` instead of `/dev/ttyS0`? **

>

> Apple Virtualization Framework uses the VirtIO Console device as the console interface of the virtual machine, and the corresponding device name is `hvc0` (Hypervisor Virtual Console). The traditional `ttyS0` is an analog 16550 UART serial port, which is less efficient.

## six Troubleshooting list

### 6.1 Startup Stage Problems

| Symptoms | Possible Causes | Troubleshooting Steps |

|-------|---------|---------|

| `Internal Virtualization error` | Incorrect kernel format | 1. Check whether the kernel needs to be decompressed<br>2. Use the `file` command to verify the format<br>3. Confirm that the file path is correct |

| No output | Console configuration error | Confirm the use of `console=hvc0` |

| Kernel panic: `not syncing` | initramfs damaged or missing | re-download/generate initramfs |

### 6.2 Root file system problems

| Symptoms | Possible Causes | Troubleshooting Steps |

|-------|---------|---------|

| `unknown-block(0,0)` | virtio driver not loaded | Confirm that initramfs contains virtio module |

| `unknown-block(254,X)` | The device exists but there is a partition error | Check the `root=` parameter |

| `No such file or directory` | Partition number error | Verify the partition layout and adjust `root=` |

| `Invalid argument` | The file system is not supported/damaged | Check the integrity of the mirror image and verify the file system type |

### 6.3 GPT-related issues

| Symptoms | Possible Causes | Troubleshooting Steps |

|-------|---------|---------|

| `Alt. header is not at the end` | GPT not repaired after disk expansion | Run `sgdisk -e disk.raw` |

| `CRC mismatch` | Partition table damage | Run `sgdisk -v` diagnosis |

### 6.4 Quick Diagnosis Command

"Bash"

# Check the kernel file format

Vmlinux file

# Check the disk image type

File disk.raw

Qemu-img info disk.raw

# Check the partition table

Fdisk-l disk.raw

Sgdisk -p disk.raw

# Verify GPT integrity

Sgdisk -v disk.raw

# View QCOW2 mirror information

Qemu-img info disk.qcow2

```

## seven. Sum up

### 7.1 Comparison of operation processes

| Steps | Ubuntu | Alpine |

| ---------------- | ------------ | --------------------------- |

| **Get the kernel** | Download → gzip decompression | Download → EFI stub extraction → gzip decompression |

| **Get initramfs** | Direct download | Direct download |

| **Get disk image** | Download cloud image → Format conversion | Recommendation: Network installation to blank disk |

| **root partition configuration** | `/dev/vda1` | Depends on the installation method |

| **Overall Complexity** | ⭐⭐ Low | ⭐⭐⭐ Medium |

### 7.2 Suggestions

1. **Ubuntu users**: Direct kernel boot unboxing experience is good, and cloud mirror compatibility is good

2. **Alpine users**:

- It is recommended to use **network installation mode** to get the best compatibility

- If you want to use cloud mirror image, you should extract the matching kernel and initramfs from the cloud mirror image.

3. **General suggestions**:

- Always verify the integrity of the downloaded file

- Remember to repair the backup header after expanding the GPT disk.

- Keep the original mirror image as a backup
